## 一、项目背景与行业痛点

### 1.1 行业与业务场景概述

随着国内文旅消费持续升级，"先查后买、先比后定"成为游客出行的标准决策路径。本项目聚焦**纯 C 端游客**的出游全链路需求，覆盖：

- **行前**：票务优惠咨询、票价计算、路线规划
- **行中**：查票、订票、订单确认
- **行中/后**：基于 RAG 的交互式景点导览、智能问答

项目以 **LangGraph + Chroma + 阿里云百炼（Qwen）** 为技术底座，定位为"游客随行的 AI 导游与票务顾问"。

### 1.2 核心行业痛点

| # | 痛点 | 现状描述 | 问题根源 | 影响 |
|---|------|----------|----------|------|
| 1 | **票务信息分散** | 同一景区在美团、携程、景区官方等多个平台价格、优惠政策、余票均不一致 | 平台数据孤岛，景区规则未结构化 | 游客决策成本高、需反复切换 App 比对，体验差 |
| 2 | **优惠政策复杂** | 老人/儿童/学生/军人/残疾人等特殊群体有不同证件要求、免票/半价规则 | 规则散落在各景区公告页，缺少统一检索入口 | 游客错过优惠、客服咨询量居高不下 |
| 3 | **导览内容粗糙** | 传统电子导览只能按固定顺序播放，不能针对"我现在站的位置"做精准讲解 | 缺乏 LLM 语义理解与 RAG 知识库 | 体验单调、个性化不足，无法回答临时问题 |
| 4 | **多轮对话能力弱** | 多数客服机器人只能一问一答，无法补全缺失参数 | 缺乏状态管理与意图路由 | 游客反复重复信息，转化率低 |
| 5 | **第三方 API 黑盒** | 票务对接涉及网络异常、限流、超时，业务代码耦合严重 | 缺少独立客户端层做兜底 | 一次网络抖动即可击穿业务流程 |

### 1.3 项目目标

| 维度 | 目标 |
|------|------|
| **业务价值** | 一站式覆盖查票/订票/优惠/路线/导览全链路；将游客决策时间从 30+ 分钟压缩到 1 分钟内 |
| **效率价值** | 7×24 在线，替代 60% 标准化票务客服工作；单次会话即可完成"问→算→订"全流程 |
| **技术价值** | 沉淀一套可复用的 LangGraph + RAG 智能体工程模板；形成 Prompt 库、工具库、客户端封装最佳实践 |

## 二、项目整体目标与范围

### 2.1 核心目标（量化）

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 业务意图识别准确率 | ≥ 90% | 通过 LangGraph 路由 + 工具调用保障 |
| 优惠政策覆盖率 | 100% | 覆盖老人/儿童/学生/军人/残疾人 5 类人群 × 3 个景区 |
| 景点导览点位 | ≥ 80 个 | 5 大景区 80+ 点位讲解词 |
| 单次响应时间 | < 3s（不含 LLM 推理） | 工具调用 + 数据查询 < 3s |
| 多轮对话参数补全成功率 | ≥ 85% | 状态机 + 追问模板 |
| 第三方调用失败率 | < 0.5% | 客户端层重试 + 异常兜底 |

### 2.2 功能范围

#### 核心模块清单

| 模块 | 关键能力 |
|------|----------|
| 意图路由 | 6 类意图：查票 / 订票 / 优惠政策 / 路线规划 / 导览 / 通用 |
| 票务工具 | `ticket_query`（多平台余票）、`ticket_book`（门票预订） |
| 优惠工具 | `verify_discount`（判定免票/半价/全价 + 所需证件） |
| 价格工具 | `calc_ticket_price`（基准价 × 折扣 + 平台服务费） |
| 路线工具 | `plan_route`（分段式游览路线 + 无障碍/老人变体） |
| 导览工具 | `guide_order_exec`（讲解 / 互动答疑 / 预订单生成） |
| RAG 知识库 | Chroma 向量库 + 6 类知识文档（票价/优惠/核验/导览/短信/订单） |
| 多轮对话 | 状态机管理 scenic_spot / visit_date / traveler_count / phone |

### 2.3 非功能要求

| 维度 | 要求 |
|------|------|
| **性能** | 首屏加载 < 5s；单轮对话 < 8s（含 LLM） |
| **稳定性** | 第三方 API 失败时自动重试 2 次；任何工具异常都要给游客友好提示 |
| **可扩展性** | 新增景区只需追加 JSON/CSV，不改代码；新增意图只需追加节点函数 |
| **可维护性** | Prompt 全部外置到 .txt 文件；日志分级输出；模块间零强耦合 |
| **可观测性** | 工具调用全量日志（参数/结果/耗时）；运行异常 ERROR 级别 |
| **安全性** | 手机号脱敏（138****5678）；API Key 走 .env，不入仓 |

### 3.5 关键设计决策

| 决策 | 备选方案 | 最终选择 | 决策理由 |
|------|----------|----------|----------|
| Agent 框架 | LangChain ReAct vs LangGraph | **LangGraph** | ReAct 输出不稳定，LangGraph 用结构化状态机和条件路由更可控 |
| 向量库 | FAISS / Milvus / Chroma | **Chroma** | 演示项目无需集群部署，本地持久化即可 |
| 前端 | Gradio / Streamlit / Flask+Vue | **Streamlit** | 30 行代码即可上线，专注对话交互 |
| 模型接入 | 直连 OpenAI / 自研网关 | **DashScope SDK** | 阿里同源、低延迟、中文场景优 |
| 客户端层 | Agent 内 HTTP | **独立 Client** | 业务代码与网络层解耦，方便 mock 与真实 API 切换 |
| Prompt 管理 | 写死在代码 / 数据库 | **TXT 文件** | 便于产品运营修改、git 可追踪 |

## 四、关键需求与核心功能实现细节

### 4.1 需求一：多轮对话与意图路由

**需求描述**  
游客可能在一句话中只说"故宫明天的票"（缺人数）、"3个人"（缺景区和日期），Agent 需自动判断缺什么并逐个追问，已获取信息不再重复索要。

**核心流程**
1. 用户消息进入 `node_parse_intent` → 调 `parse_user_info` 工具解析
2. 提取 `core_need` → 映射到 6 类意图（`ticketing` / `order_generation` / `policy_inquiry` / ...）
3. 进入 `node_collect_info` → 状态机检查 `scenic_spot / visit_date / phone` 缺失项
4. 一次只追问一个字段，更新 `collect_step`
5. 全部补齐后路由到对应业务节点

**关键代码**
```python
class TravelState(TypedDict):
    messages: Annotated[list, add_messages]
    scenic_spot: str
    visit_date: str
    traveler_count: int
    phone: str
    intent: str
    missing_fields: list
    collect_step: int
    final_answer: str

# 状态机 + 条件边
graph = StateGraph(TravelState)
graph.add_node("parse_intent", node_parse_intent)
graph.add_node("collect_info", node_collect_info)
graph.add_conditional_edges("collect_info", should_collect_more, ...)
```

**效果**  
参数补全成功率 ≥ 85%，用户平均对话轮次 2.3 轮完成一次订票。

| # | 工具 | 入参 | 关键逻辑 |

- 手机号统一脱敏：`phone[:3]}****{phone[-4:]}`

### 4.3 需求三：RAG 检索与内容总结

**需求描述**  
当游客问"60 岁老人去故宫要带什么证件"，需要从 `data/knowledge/特殊群体优惠政策.txt` 中精准检索并生成回答。

**核心流程**
```
用户提问 → Embedding → 向量库 Top-K 召回
         ↓
   PromptTemplate（rag_summarize）组装 input + context
         ↓
   ChatTongyi 总结 → StrOutputParser 解析
         ↓
   返回自然语言答案
```

**关键代码**
```python
class RagSummarizeService:
    def __init__(self):
        self.vector_store = VectorStoreService()
        self.retriever = self.vector_store.as_retriever(search_kwargs={"k": 3})
        self.prompt_text = load_rag_prompts()
        self.prompt_template = PromptTemplate.from_template(self.prompt_text)
        self.chain = self.prompt_template | self.model | StrOutputParser()

    def rag_summarize(self, query: str) -> str:
        context_docs = self.retriever_docs(query)
        context = ""
        for i, doc in enumerate(context_docs, 1):
            context += f"【参考资料{i}】: {doc.page_content} | {doc.metadata}\n"
        return self.chain.invoke({"input": query, "context": context})
```

**关键技术点**
- **MD5 增量入库**：文件未变不重新向量化，节省 API 费用
- **DashScope 限流**：每次最多 10 条，分批入库
- **Prompt 强约束**："回答必须完全基于参考资料，不编造"

**效果**  
6 类知识库、Top-3 召回、回答事实准确率 95%+。

### 4.4 需求四：Prompt 工程与提示词管理

| Prompt | 角色 | 核心约束 |

### 4.5 需求五：日志系统设计

**核心要求**
- 控制台 INFO、文件 DEBUG 双输出
- 按天滚动日志文件 `agent_20250610.log`

**实现**

```python
def get_logger(name="agent", console_level=INFO, file_level=DEBUG):
    logger = logging.getLogger(name)
    if logger.handlers:        # 关键：去重
        return logger
    console_handler = StreamHandler()
    file_handler = FileHandler(f"logs/{name}_{date}.log", encoding="utf-8")
    logger.addHandler(console_handler)
    logger.addHandler(file_handler)
    return logger
```

**踩坑点**  
Streamlit 文件变更会重载模块，logger 被多次实例化会重复输出。**`if logger.handlers: return logger`** 这一行就是防御。

---

### 4.6 需求六：第三方票务客户端封装

**核心要求**
- 网络异常自动重试 2 次
- 超时 10s 兜底
- 失败抛 `TicketClientError`，Agent 层捕获后转友好提示
- `base_url` 为空时使用内置 mock 数据，便于演示

**实现**
```python
class TicketClient:
    def query_tickets(self, scenic_spot, visit_date):
        attempt = 0
        while attempt <= self.retry:
            try:
                # TODO: 接入真实 API
                return self._mock_query(scenic_spot, visit_date)
            except Exception as e:
                attempt += 1
                last_error = e
        raise TicketClientError(f"查询失败: {last_error}")
```

**架构价值**  
Agent 层零网络依赖，未来切换真实 API（如美团开放平台）只需改 Client，业务代码零改动。

## 六、遇到的难点与解决方案

### 6.1 技术难点

| # | 难点 | 现象 | 解决方案 |
|---|------|------|----------|
| 1 | **LLM 输出不稳定** | ReAct Agent 有时输出格式错误，解析失败 | 切换到 LangGraph 状态机，用结构化字段而非自由文本 |
| 2 | **RAG 召回不准** | 用户问"老人优惠"但召回的是"儿童优惠" | 优化 chunk_size=200、overlap=20；用 `text-embedding-v4` 替代老版本 |
| 3 | **Embedding 限流** | DashScope 每次最多 10 条 | 分批入库 `batch_size=10` |
| 4 | **MD5 增量失效** | 文档改了 MD5 变，重新入库 | 维护 `md5.txt` 记录已入库文件哈希 |
| 5 | **多轮上下文丢失** | 用户第 3 轮时 Agent 忘记前文信息 | LangGraph State + `add_messages` 自动维护消息列表 |
| 6 | **手机号泄露** | 订单回显明文手机号 | 客户端层统一脱敏 `138****5678` |

### 6.2 踩坑记录

| # | 现象 | 根因 | 修复 |
|---|------|------|------|
| 1 | 日志在 Streamlit 中重复输出 | Streamlit 文件热重载触发模块重载 | `if logger.handlers: return logger` |
| 2 | Chroma 切换 Embedding 模型报错 | 向量维度不一致 | 删除 `chroma_db/` 目录重新入库 |
| 3 | API Key 提交到 Git | 写死在代码里 | 改用 `.env` + `python-dotenv` |
| 4 | Prompt 中 `{}` 被误解析 | `order_output_prompt` 有大量 JSON 示例 | 含示例的 Prompt 用 `str.format` 而非 `PromptTemplate` |
| 5 | 路径在不同目录运行报错 | 相对路径 | 统一封装 `get_abs_path()` |
| 6 | qwen-turbo 上下文超长 | Prompt 拼接 messages 超 8K | 切换到 `qwen-plus`（32K）或精简历史 |
| 7 | `langchain_community.embeddings.DashScopeEmbeddings` 弃用警告 | 新版本改至 `langchain_qwen` | 暂保留旧版（功能正常），后续升级 |

### 6.3 优化思路

- **流式输出**：未来引入 `ChatTongyi(streaming=True)`，首 token < 500ms
- **缓存层**：相同 query 命中后直接返回，节省 LLM 调用费用
- **Prompt A/B**：建立多版本 Prompt 对照组，数据驱动调优
- **链路追踪**：集成 LangSmith，可视化 Agent 执行路径

## 八、总结与个人收获

### 8.1 项目整体复盘

本项目从零搭建了一套**生产可用的 C 端文旅智能体**，核心价值不在于"做出能聊天的机器人"，而在于：

1. **完整业务闭环**：从问优惠 → 算价格 → 查余票 → 下订单 → 导览讲解，链路无断点
2. **工程化落地**：所有 Prompt/配置/工具均可独立维护和扩展
3. **架构可复用**：分层思想适用于任何垂直领域 Agent

**最关键的认知**：Agent 的核心竞争力不是"用了多牛的模型"，而是**对业务场景的拆解深度**和**对工程质量的把控**。

### 8.2 技术与业务认知提升

- **认知 1**：LangGraph ≠ LangChain ReAct。结构化状态机比自由推理更可控
- **认知 2**：RAG 不是"塞向量库就完事"，chunk_size / overlap / 提示词约束缺一不可
- **认知 3**：Prompt 是产品的一部分，要像代码一样 review 和版本管理
- **认知 4**：Agent 层严禁直连外网，独立 Client 层是工程纪律
- **认知 5**：演示项目也能用生产级架构，关键是分层的"形状"对不对

### 8.3 可迁移经验

- ✅ **垂直领域 Agent 落地三板斧**：业务拆解 → 工具抽象 → 状态机路由
- ✅ **Prompt 外置四件套**：主提示词 + RAG 总结 + 业务输出 + 路径配置
- ✅ **客户端层封装模式**：重试 + 超时 + 兜底 + mock 切换
- ✅ **日志去重技巧**：`if logger.handlers: return logger`
- ✅ **MD5 增量入库**：节省 Embedding 费用
- ✅ **状态机 + 追问模板**：多轮对话的工程化范式

### 8.4 给后续学习者的建议

1. **先跑通，再优化**：第一章先把环境跑起来，比读 10 篇博客有用
2. **逐行复刻**：本项目设计为"边学边抄"，不要一上来就改架构
3. **关注分层而非细节**：理解"为什么这么分"比"这一行干嘛"更重要
4. **业务先行**：先想清楚 6 类意图是什么，再去看 LangGraph 怎么实现
5. **保持怀疑**：所有 Prompt 模板都只是起点，要用真实数据调优
