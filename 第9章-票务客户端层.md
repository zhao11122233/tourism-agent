## 二、核心知识点讲解

### 2.1 为什么 Agent 层不能直接发网络请求？

```
Agent 层（agent_tools.py）
    ↓ 只调用 TicketClient，不 import urllib/requests
TicketClient（client/ticket_client.py）
    ↓ 封装 HTTP 请求、超时、重试、鉴权、异常
第三方票务 API（美团/携程/景区官方）
```

- **职责分离**：Agent 层负责"对话和意图"，Client 层负责"网络和数据"
- **可替换**：换一家票务供应商，只改 Client 不改 Agent
- **可测试**：Agent 层单元测试时 mock TicketClient 即可

### 2.2 模拟数据 vs 真实 API

当前项目运行在"模拟模式"（`base_url` 为空），`_mock_query` 和 `_mock_book` 返回模拟数据。接入真实 API 只需：
1. 在 `config/agent.yml` 中填入 `base_url`
2. 实现 `query_tickets` / `book_ticket` 中的 HTTP 请求逻辑
3. 模拟代码不动，作为回退方案

### 2.3 重试机制

```python
attempt = 0
while attempt <= self.retry:  # retry=2，最多尝试 3 次
    try:
        result = self._mock_query(...)
        return result
    except Exception as e:
        attempt += 1
        if attempt <= self.retry:
            time.sleep(1)  # 等 1 秒再重试
raise TicketClientError(f"失败（已重试{self.retry}次）")
```

- 简单重试 + 固定 1 秒间隔
- 适合学习场景；生产环境可升级为指数退避（1s → 2s → 4s）

### 2.4 自定义异常

```python
class TicketClientError(Exception):
    """票务客户端异常"""
```

优于直接用 `Exception`：调用方可以 `except TicketClientError` 精确捕获票务相关异常，与代码 bug 导致的异常区分开。

## 五、验收标准+踩坑避坑点

### 5.1 验收标准
- [ ] `TicketClient().query_tickets("故宫", "2025-06-15")` 返回 3 个平台的余票数据
- [ ] 旺季（7 月）和淡季（1 月）查询同一景区，价格不同
- [ ] 不存在的景区返回 fallback 价格（80 元）且不崩溃
- [ ] `book_ticket` 手机号校验生效：非 11 位报错

### 5.2 踩坑避坑点
1. **不要从 agent_tools 导入 TicketClient**：那是反向依赖。正确方向是 `agent_tools → ticket_client → data/pricing/scenic_spots.json`
2. **模拟数据有随机性**：`random.randint(0, 30)` 导致每次查询余票数不同，这是设计如此（模拟真实票务波动），不是 bug
3. **base_url 为空不等于出错**：这是一个"特性开关"，为空走模拟模式是正常流程
4. **重试次数是 while <= retry**：retry=2 意味着"首次 + 2 次重试 = 最多 3 次"，不是只重试 2 次
